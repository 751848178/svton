## 任务 1：腾讯云 CDN provider 改用官方 SDK

**问题**：`cdn-config/providers/tencent-cdn-provider.ts:45-46` 的 TC3-HMAC-SHA256 签名留空（`Signature=` 空串），且把 secretKey 明文放进 `X-TC-Secret-Key` header。安全 + 功能双重缺陷。

**前提**：`tencentcloud-sdk-nodejs-cdn@4.1.250` 已在 package.json，但 node_modules 未安装。实现前需先 `pnpm install`。

**实现步骤**：
1. 重写 `tencent-cdn-provider.ts`，用 `tencentcloud-sdk-nodejs-cdn` 的 `CdnClient`：
   - `require('tencentcloud-sdk-nodejs-cdn').cdn.v20180607.CdnClient`（实现时需在装好依赖后确认确切路径）
   - `PurgeUrlsCache({ Urls })` / `PurgePathCache({ Paths })`，SDK 内部完成 TC3 签名
   - 删除 `HttpService` 注入、`buildTencentHeaders`、`X-TC-Secret-Key` 明文 header
   - 返回 `{ requestId: response.RequestId }`
2. 保持 `CdnRefreshProvider` 接口契约不变（`purge(creds, urls, isDirectory)`）
3. 更新现有 provider 单测：mock SDK 模块而非 HttpService
4. **验证**：实现时如果 SDK 导出结构与 node 内联检查不一致，回退到"补完手写 TC3 签名"方案（按官方规范完整实现 Date/CanonicalRequest/StringToSign/Signature 四层 HMAC），不返回空签名

**风险/缓解**：注释说"SDK TS 类型不全" —— 若确实如此，用 `require` 动态加载 + `any` 类型，不引入 `@types` 依赖。功能正确性优先于类型完美。

## 任务 2：CBC 凭据加密 KDF 强化

**问题**：`common/crypto/crypto.constants.ts:32-35` 的 `deriveCbcKey` 不做 KDF，只 `padEnd(32).slice(0,32)`。弱密钥派生。

**约束**（用户已选 scryptSync 方案）：直接改 KDF 会让历史 CBC 密文解不开（key 变了）。需要双密钥策略：旧密钥解密历史数据，新密钥加密新数据，并在读取时尝试两种 key。

**实现步骤**：
1. 在 `crypto.constants.ts` 新增：
   - `CBC_LEGACY_SALT`（保留历史行为标识）
   - `deriveCbcKeyLegacy(envKey)`：当前实现（截断/pad），仅用于解密历史密文
   - `deriveCbcKey(envKey)`：改为 `crypto.scryptSync(envKey ?? CBC_DEFAULT_KEY, CBC_SALT, CBC_KEY_LENGTH)`，新增 `CBC_SALT = 'cbc-salt'`
2. 在 `CryptoService` 持有两个 cbc key：
   - `cbcKey`（新，scrypt）→ 用于 `encryptCbc`
   - `cbcKeyLegacy`（旧，截断）→ 解密时回退
3. `decryptCbc` 改为先用新 key 解密，失败（throw）则回退到 legacy key；两次都失败才抛错。这样：
   - 历史密文：新 key 失败 → legacy key 成功 → 解密正常
   - 新密文：新 key 直接成功
   - 过渡期后可删 legacy（留 TODO）
4. `encryptCbc` 保持用新 key（scrypt）
5. **不**改 `resource-pool-credential.utils.ts` 的 `deriveCbcKey` 调用 —— 它注释说只服务测试，且现在仍调 `deriveCbcKey`（会自动变 scrypt），其测试如果断言 wire 格式需同步更新
6. **测试**：
   - `crypto.service.spec.ts` 加 case：旧 key 加密的密文能用新代码解出
   - 更新 `resource-pool.service.spec.ts:70`、`admin.service.spec.ts:38`、`project-environment.service.spec.ts:62` 的 decrypt 调用路径（它们走 `decryptCbc`，自动受益）
   - 加一个 fixture：用旧 padEnd key 预先加密一段密文常量，断言新代码能解开（回归保护）
7. **CBC 直连调用点**（4 处，非 crypto.service）：`resource-pool.service.ts:39`、`resource-pool-allocation-lifecycle.service.ts:59`、`key-center.service.ts:18`、`project-environment-resource-copy.service.ts:174` —— 这轮**不**改它们走 GCM（用户没选 GCM 迁移），它们继续调 `encryptCbc`，但加密的密文现在用 scrypt key

**风险**：双 key 解密的 try/catch 要区分"key 不对"和"密文格式损坏"——只在格式合法但 final 失败时回退，格式错误（split 后缺段）直接抛错。

## 任务 3：Cloudflare provider 改用官方 SDK

**问题**：`cloudflare-cdn-provider.ts:24-31` 用 raw HttpService 调 Cloudflare API。bearer token 本身没错，但缺 SDK 的重试/错误模型。

**前提**：`cloudflare@6.5.0` 已在 package.json，node_modules 未安装。

**验证事实**（已确认）：SDK 导出 `{ Cloudflare, APIError, RateLimitError, ... }`，构造函数 `new Cloudflare({ apiToken })`。

**实现步骤**：
1. 重写 `cloudflare-cdn-provider.ts`：
   - `import { Cloudflare } from 'cloudflare'`
   - `const client = new Cloudflare({ apiToken })`
   - `await client.zones.purgeCache(zoneId, { files: urls })` 或 `{ purge_everything: true }`
   - 删除 `HttpService`、`baseUrl`
   - 返回 `{ requestId: data?.result?.id }`
2. 更新单测：mock `cloudflare` 模块
3. 接口契约（`CdnRefreshProvider`）不变

**风险**：Cloudflare SDK 的 `zones.purgeCache` 精确签名实现时核对（是 `purgeCache(zoneId, params)` 还是 `zones(zoneId).cache.purge()` —— SDK 版本间有差异）。若签名不符，回退保留当前 raw HTTP 实现（bearer 鉴权正确，只是缺 SDK 健壮性，可降级为非阻塞技术债）。

## 任务 4：certbot/openssl stdout 解析替换为结构化解析

这是两个独立子任务，工作量和风险都最大。

### 4a：openssl 证书解析改用 `node:crypto` 的 X509Certificate

**问题**：`site-tls-openssl-parser.utils.ts:8-38` 按行 split + 正则匹配 `openssl x509 -text` 的文本输出，locale/版本敏感。

**关键风险**（已验证）：`server-executor/adapters/ssh-live-json.utils.ts:9` 在 **8000 字符截断** stdout。改 PEM 解析后，完整 PEM 链（含链式证书）可能接近这个长度。**必须保留文本解析作为截断回退**。

**实现步骤**：
1. 修改 `buildSiteTlsProbeCommand`（`site-tls-probe.ts:51`），命令改为输出 PEM：
   - `echo | openssl s_client -servername ${host} -connect ${host}:${port} 2>/dev/null | openssl x509 -noout` → 改为只取 PEM 的形式
   - 实际改为：`... | openssl x509`（不加 `-noout`，输出完整 PEM）。但 PEM 较长且易被 8000 字符截断
   - **更稳妥**：保留现有命令（输出 `subject=/issuer=/notAfter=` 键值对），同时**额外**解析 PEM 若可用。即两条命令合一条：先输出键值对（紧凑、抗截断），再可选输出 PEM
2. 新增 `parseX509FromPem(pemText)` util（在 `site-tls-openssl-parser.utils.ts`）：
   - 用 `new crypto.X509Certificate(pemBuffer)`
   - 提取 `subject`、`issuer`、`serialNumber`、`validFromDate`（notBefore）、`validToDate`（notAfter）、`fingerprint256`
   - 带返回类型与现有 `parseOpenSslCertificateText` 对齐
3. `extractSiteTlsProbeMetadata`（`site-tls-probe.ts:55`）改为：优先尝试 PEM 解析，失败则回退到文本解析
4. `parseOpenSslDate` 保留（PEM 解析出的日期是 Date 对象，不需要它；但回退路径需要）—— 或新增 `parseX509Date` 用 X509Certificate 直接给的 Date
5. **测试**：`site-tls-probe` 相关 spec 加 PEM fixture case（用真实的 self-signed cert PEM 常量）
6. **回退保证**：PEM 解析失败（截断/格式坏）→ 回退到现有文本解析，零回归

**风险**：`buildSiteTlsProbeCommand` 的命令改动会影响所有调用点（`site-ops-plan.utils.ts:131`、`server-executor-site-tls-follow-up.utils.ts:41`）。命令改动要谨慎 —— 建议只**追加** PEM 输出段，不删除现有键值对段。

### 4b：certbot 结果分类弱化 stdout 正则依赖

**问题**：`site-certbot-classifier.utils.ts:20-32` 用 `/congratulations/`、`/successfully renewed/` 等 English stdout 正则分类，locale/版本敏感。

**关键事实**（已验证）：`classifyCertbotRenewOutput(text, executionStatus)` **已经接收 `executionStatus` 参数**（`'completed'`/`'failed'`），当前实现 L16/L32 已部分用它，但主分类逻辑（L20-30）仍以 stdout 正则为准。

**实现步骤**（重构而非引入新依赖）：
1. 重排 `classifyCertbotRenewOutput` 的判定优先级：
   - `executionStatus === 'failed'` → `failed`（已如此，L16）
   - `executionStatus === 'completed'` 且无错误关键词 → `succeeded`（新：把退出码/状态作为主信号）
   - stdout 正则降级为**辅助**：仅用于区分 `not_due`（`no renewals were attempted` 这种非错误跳过）vs `succeeded`，以及提取 `summary` 文本
2. `selectSummaryLine` 保留 —— 它只做摘要提取（从 stdout 找一行描述），这本身不脆弱，因为 summary 是给人看的，分错类才致命
3. **不**引入 certbot JSON 输出解析（certbot 的 `--deploy-hook`/JSON 输出需要改命令，影响面大，且 certbot certonly/renew 的 JSON 支持版本不一）—— 这轮只做"退出码为主、stdout 为辅"的重排
4. **测试**：新增 case：空 stdout + `executionStatus='completed'` → `succeeded`（当前会落到 `unknown`/`failed`，是 bug）；stdout 只有 locale 中文 + completed → succeeded

**风险**：certbot 真实退出语义 —— `certbot renew` 退出码 0 = 全部成功或无需续期；非 0 = 至少一个失败。当前 `executionStatus` 是否精确映射 certbot 退出码需确认（`executionStatus='completed'` 可能来自 SSH exit 0）。如果 `completed` 严格对应 exit 0，那么主信号化是安全的。
