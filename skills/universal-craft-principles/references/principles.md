# Universal Craft Principles

## First choices

当一段通用业务代码开始失控，优先回答这三个问题：

1. **稳定契约是什么**：调用方究竟应该知道哪些输入和输出。
2. **谁在编排，谁在执行**：入口层是不是背着不该有的底层细节。
3. **复杂度应该藏在哪**：fallback、重试、类型分支、供应商差异应不应该被调用方感知。

## Core principles

### 1. Contracts first

- 先固定入口函数、请求类型和结果类型，再重排内部职责。
- 调用方应面对稳定接口，而不是条件分支菜单。
- 一次重构里，优先让契约更清楚，再让实现更优雅。

### 2. Module topology

- 一个模块只承担一种职责意图。
- 对等实现优先并列 handler、provider、service，不挤在一个分支巨兽里。
- 入口层做 dispatcher shell，执行层做真实工作。

### 3. Service-owned complexity

- fallback、重试、源切换、异常包装属于服务层，不属于调用方。
- 新增一种供应商或任务类型时，改动面应该主要落在并列单元。
- 调用方知道得越少，扩展面通常越健康。

### 4. State economy

- 可计算值优先派生，不复制成长期维护的冗余字段。
- 如果多个地方都在同步同一业务事实，通常说明归属错了。
- 结果对象可以集中组装，但事实来源应尽量唯一。

## Anti-pattern translations

- `入口函数越来越长`：多半不是该抽更多私有函数，而是该拆编排与执行职责。
- `调用方要 try primary / catch fallback`：说明服务层还没把策略收回去。
- `新增一种类型要改五处`：缺少并列分发结构或稳定契约。
- `很多字段到处传`：先判断是不是只有一个地方真的拥有这些事实。

## Success criteria

- 调用方只面对一个更稳定的接口和更少的分支知识。
- 新增一种实现时，主要通过增加并列单元完成，而不是修改一串旧分支。
- 模块名字更像角色和职责，而不是流程片段或历史兼容层。
