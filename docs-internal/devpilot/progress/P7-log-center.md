# P7. Log Center Progress

## Goal

`LogCenter` is the project/environment-scoped observability surface for log
streams, archived log entries, collection runs, SSE tail sessions, source
metadata, Server executor follow, SLS read-only ingestion, and future agent-level
continuous follow runtime.

## Current Status

- Roadmap status: log streams, archived entries, collection runs, SSE tail,
  bounded session governance, Server executor scheduled follow, SLS
  credential-backed ingestion, Agent Follow metadata persistence contract,
  read-only server-agent task-pull log follow sample hints, default-off
  server-agent task-pull claim/ack/terminal writeback boundaries, and minimal
  log collection finish sync exist in the current control plane.
- Remaining product/runtime gaps from the source docs: real agent-level
  continuous log follow runtime after terminal writeback, non-log business run
  result sync, broader provider-backed log sources, and deeper production
  scheduling/e2e coverage remain follow-ups.
- Recent verified structure slices: F129 repaired the Agent Follow metadata
  type/helper contract; F138 split `use-logs-tail.ts` metadata and stream
  effects into focused hooks, keeping those hook files under 200 lines.
- Current source-backed structure result: F149 split stream CRUD, entry append,
  collection run, and retention cleanup mutations into
  `use-logs-actions.hooks.ts`; `use-logs.ts` now owns loading, derived target
  options, selected stream composition, and tail/policy/action hook composition.
- Current runtime-contract repair: F182 wires scheduled Agent Follow metadata
  into the existing Server executor collection path with
  `requiredTransport=server_agent` and blocks unsafe SSH fallback when the
  resolved target is not `server_agent`.
- Current backend structure result: F241 split Log Collection ingestion
  persistence into `LogCollectionIngestionRepository`; the service now owns
  completed-run gating, extraction, parsing, and redaction orchestration while
  the repository owns Prisma reads/writes.
- Current DTO structure result: F242 split the over-limit Log Center DTO barrel
  into stream, entry, and collection DTO files while preserving the existing
  `./dto/log-center.dto` import contract.
- Current scheduler structure result: F243 moved Server/Agent Follow collect
  params, scheduler config reads, and empty-summary construction into focused
  helpers, leaving the scheduler focused on scanning, throttling, and collection
  execution.
- Current SLS scheduler structure result: F244 moved SLS backfill metadata
  policy, collect params, scheduler config reads, and empty-summary construction
  into focused helpers, leaving the scheduler focused on scanning, throttling,
  and collection execution.
- Current SLS adapter structure result: F245 moved Aliyun SLS live query types,
  row parsing/formatting, provider-request policy, and result envelope building
  into focused helpers, leaving the adapter focused on credential gating, SDK
  loading, and paged GetLogs orchestration.

  collection planning, SSE tail, Server/Agent Follow metadata, and copy helpers
  through hook fields. Completed collection runs are ingested through a focused
  service/repository boundary that preserves project/environment-scoped entry
  context and stream snapshots. Backend `/logs` routing is split by base
  stream/run/stats, stream operation, and tail/session route groups while
  preserving project/environment access checks before mutation or stream reads.
  Append, collect, and retention audit payloads now flow through a focused audit
  writer service without changing the user-facing mutation result.
  Collection-run execution orchestration is now isolated from the public facade,
  and stream-scoped write lookup/guards now live behind a focused orchestration
  service while preserving the same collect/update/append/retention entrypoints,
  while server-source command-plan generation, Server executor execution, and SLS
  provider collection keep their existing service boundaries. Stream list/access/lookup reads and stream create access
  target resolution remain project/environment scoped and are now isolated behind
  focused service facades. Collection/retention run list reads are now isolated
  while keeping run creation/update/cleanup execution stable. Provider-backed
  collection planning remains read-only and is now isolated from collection
  orchestration. Entry list/tail/stats reads are now isolated while keeping
  cursor semantics and stats windows stable.
  Retention cleanup execution is now isolated while keeping `/logs` run results
  stable. Manual append-entry execution is now isolated while preserving
  stream-scoped redaction and project/environment entry context. Stream create/
  update mutation payload shaping is now isolated while preserving access-policy
  flow.
- Organization map: `page.tsx` is the route shell; `use-logs.ts` is the page data
  composition hook; `use-logs-tail.ts` composes metadata, polling, and SSE stream
  effect hooks for tail session lifecycle; components render stream forms, stream cards,
  tail controls, entries, collection runs, and metadata/follow controls. Backend
  controllers now keep HTTP/SSE entrypoints separate from shared log access
  filtering, stream-session audit writes, and SSE limit normalization helpers.
  `LogCenterService` keeps orchestration, while `LogCenterAuditService` owns
  audit payload construction and pure utils own entry queries, tail cursors, and
  provider/value normalization. A focused server collection plan utility owns

  `LogServerCollectionExecutionService` owns Server executor target resolution,
  Agent Follow transport blocking, queue/live submission, and result
  normalization; `LogCollectionRunExecutionService` owns collection run
  creation/update, server/provider dispatch, completed-run ingestion, final run
  reload, and collection audit orchestration; `LogStreamWriteOrchestrationService`
  owns stream-scoped collect/update/append/retention lookup and archived guards;
  `LogEntryQueryService` owns entry list/tail/stats read
  execution and response shaping; `LogStreamQueryService` owns stream
  list/access/lookup reads; `LogRunQueryService` owns collection/retention run
  list reads; focused target-context services own

  collection plan service/types/utils own unsupported-provider and SLS dry-run/
  live plan shaping. `LogRetentionCleanupService` owns retention run
  execution and stream last-entry refresh after live deletion.
  `LogEntryAppendService` owns normalized append persistence, redaction, stream
