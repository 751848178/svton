P6 covers Devpilot monitoring and alerting control-plane capability for
resource metrics, service SLO visibility, alert rules/events, silences,
notification channels, deliveries, retries, and escalation governance.

  components, types, constants, and utils are all below 200 lines.
- Source-backed API structure gap: `monitoring.service.ts` remains over the

  that re-exports focused DTO input files, `monitoring-scheduler.service.ts`

## Maps To Maintain

- Business logic map: monitoring API receives query/body DTOs for alert rules,
  alert events, dashboards, silences, notification channels, and notification
  deliveries; `MonitoringService` performs the actual rules, dashboards,
  delivery, retry, and escalation logic.
- Organization map: focused monitoring controllers and `MonitoringService`
  currently import DTO classes through `./dto/monitoring.dto`; focused DTO
  files own domain-specific input classes, while `monitoring.dto.ts` remains
  the stable public barrel. `MonitoringAccessService` centralizes controller
  read/write access-policy wrapping. `MonitoringSchedulerService` owns
  scheduler orchestration; `MonitoringSchedulerConfigService` owns scheduler
  env/config interpretation; scheduler summary types and default summary value
  construction live in focused scheduler files. The resource metric dashboard
  and service SLO dashboard read-models now live behind focused query/builder
  services while `MonitoringService` preserves the existing public method
  contracts for dashboard callers and service SLO evaluation. Notification
  delivery read/access-scope queries now live behind a focused read service.
  Notification delivery payload/text/email formatting now lives behind focused
  builder/types/utils boundaries. Notification delivery dispatch/config/write
  execution now lives behind focused dispatch, adapter, writer, and config
  services, notification retry orchestration/audit now lives behind focused
  retry services, and alert escalation scan/dedupe/audit now lives behind
  focused escalation services. `MonitoringService` remains the compatibility
  delegate for public monitoring APIs and still owns alert rule/evaluation
  orchestration pending later slices.
- Functional map: alert rules/evaluation, alert events, resource metric
  dashboard, service SLO dashboard, alert silences, notification channels, and
  notification deliveries are the current controller/DTO groups.
- Data-flow map: HTTP request/query/body -> class-validator DTO -> controller
  policy/scope checks -> `MonitoringService` operations -> existing response
  shapes. Dashboard controller -> `MonitoringService` compatibility delegates
  -> focused resource metric or service SLO dashboard query service -> Prisma
  signal reads -> focused builder/status services -> existing response shapes.
  Notification controller -> `MonitoringService` compatibility delegates ->
  focused notification delivery read service -> Prisma delivery/channel/event
  reads -> controller access filtering or retry access check.
  Notification dispatch/retry/escalation paths -> `MonitoringService`
  orchestration -> focused notification dispatch service -> Webhook/Email
  adapter services -> payload builder/config service/writer service -> existing
  delivery and channel status response shapes.
  Default-off scheduler tick -> `MonitoringSchedulerConfigService` flags/bounds
  -> Prisma alert-rule scan and notification/escalation queries ->
  `MonitoringService` evaluation/retry/escalation operations -> scheduler
  summary result.
- Page-structure map: `/monitoring` Web page consumes monitoring APIs through
  `useMonitoring()`, then renders dashboard, rule, event, silence, and channel
  panels; this slice does not change the page UI.
